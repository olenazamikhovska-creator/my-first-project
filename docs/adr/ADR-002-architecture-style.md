# ADR-002: Вибір монолітної архітектури для інтернет-магазину

## Статус
Accepted

## Контекст
Система інтернет-магазину розрахована на ~200 одночасних користувачів, ~5000 замовлень на добу. 
Команда розробки складається з 3-4 осіб. Термін реалізації проєкту — 4-6 місяців. 
Потрібна швидка розробка MVP з можливістю масштабування в майбутньому. 
Складність бізнес-логіки: середня (каталог товарів, кошик, замовлення, оплата).

## Рішення
Використовувати **монолітну архітектуру** (Monolithic Architecture) з чітким поділом на шари:
- Presentation Layer (Frontend: React/Vue)
- Application Layer (Backend API: Node.js/Express)
- Domain Layer (Бізнес-логіка)
- Infrastructure Layer (Database, External Services)

## Обґрунтування
1. ✅ **Швидкість розробки**: моноліт дозволяє швидко ітерувати та вносити зміни без необхідності синхронізації між сервісами
2. ✅ **Простота розгортання**: один додаток замість координації десятків мікросервісів; достатньо одного сервера на старті
3. ✅ **Простота тестування**: інтеграційні тести запускаються швидше, немає мережевих затримок між сервісами
4. ✅ **Менша складність**: для команди з 3-4 осіб мікросервіси створять зайвий overhead на підтримку інфраструктури
5. ✅ **ACID-транзакції**: простіше забезпечити цілісність даних в межах одного додатка

## Альтернативи
| Варіант | Чому відхилено |
|---------|---------------|
| **Мікросервіси** | Занадто складно для поточного масштабу; потрібна окрема команда DevOps; overhead на мережеві виклики; складність distributed transactions |
| **Serverless (AWS Lambda)** | Vendor lock-in; холодний старт критичний для інтернет-магазину; складність локальної розробки та тестування; вища вартість при стабільному навантаженні |
| **SOA (Service-Oriented Architecture)** | Надмірна складність для проєкту; потрібна ESB або service mesh; команда не має досвіду з розподіленими системами |

## Наслідки
### ✅ Позитивні
- Швидкий старт розробки (2-3 тижні до першого робочого прототипу)
- Просте розгортання на одному VPS сервері
- Легше знаходити та виправляти баги (один кодбейз)
- Можливість виділити мікросервіси пізніше, коли система виросте

### ⚠️ Негативні
- При зростанні до >50,000 замовлень/добу знадобиться рефакторинг
- Всі компоненти масштабуються разом (неможливо масштабувати тільки "вузьке місце")
- Триваліший час збірки при великій кодовій базі
- Ризик "God Object" — потрібно суворо дотримуватися шаруватої архітектури
